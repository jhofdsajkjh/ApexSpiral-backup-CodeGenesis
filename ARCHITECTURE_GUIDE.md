# 分层架构与模块化拆分指南

## 核心架构思想

自上而下：需求梳理 → 整体架构 → 模块拆分 → 逻辑分层 → 细节编码

## 标准四层架构

1. **接入层**：接口请求、参数接收、路由分发
2. **业务层**：核心业务逻辑、流程控制、规则判断
3. **工具层**：公共方法、算法封装、工具函数、通用能力
4. **底层层**：配置常量、数据读写、底层依赖、基础支撑

## 模块化拆分原则

1. 单一职责：一个模块只负责一项核心功能
2. 低耦合：模块之间依赖少、互不干扰
3. 高内聚：同类逻辑集中存放，不分散散落
4. 可扩展：预留扩展接口，后续新增功能无需重构原有代码

## 目录规范标准

- /config 配置常量
- /utils 公共工具
- /core 核心逻辑
- /api 接口服务
- /test 单元测试
- /docs 注释文档

## 最优工程路线（融合 agent-skills）

### 标准六阶段：DEFINE → PLAN → BUILD → VERIFY → REVIEW → SHIP

1. **规范先行 /spec**：定需求、接口、边界、风格
2. **任务拆解 /plan**：原子任务、验收标准、依赖顺序
3. **增量构建 /build**：薄切片纵向实现、小步提交
4. **测试验证 /test**：自动测试、调试、验证
5. **质量评审 /review**：多角色自检、安全性能规范
6. **安全上线 SHIP**：可监控、可回滚
